# HW5: оптимизация приложения по скорости и памяти

## Проблемы исходной версии

- Для каждой строки из `words.txt` выполнялся отдельный `requests.get(...)`.
- Для каждого слова HTML заново разбивался через `text.split()`.
- В `words.txt` находится `7866` строк, но уникальных слов только `93`.
- В результате приложение делало `7866` сетевых запросов и тысячи раз повторяло одну и ту же тяжёлую работу.

## Что было сделано

1. Чтение слов вынесено в отдельную функцию `load_words()`, где входные данные дедуплицируются с сохранением порядка.
2. Вместо тысяч HTTP-запросов приложение теперь делает один запрос за весь запуск.
3. Подсчёт частот переписан на один проход по данным.
4. Для сетевого сценария добавлена потоковая обработка HTML через `iter_content(...)`, чтобы не создавать заново большие временные структуры.
5. Для локального профилирования добавлена функция `count_word_frequencies_in_text(...)`, которая считает слова лениво через regex и не делает полного `split()` по всему документу.
6. Добавлен бенчмарк `benchmark.py`, который сравнивает исходный алгоритм и оптимизированную версию на одном и том же HTML.
7. Добавлены тесты `test_main.py`, которые проверяют корректность токенизации и совпадение результата с исходным алгоритмом.
8. Добавлен `page_snapshot.html`, чтобы профилирование было воспроизводимым и не зависело от нестабильности внешней сети.

## Результат оптимизации

Бенчмарк запускается на полном наборе из `words.txt` и сравнивает исходный алгоритм с новой реализацией.

- Строк в `words.txt`: `7866`
- Уникальных слов: `93`
- HTTP-запросов в исходной версии: `7866`
- HTTP-запросов после оптимизации: `1`
- Время исходной версии: `0.944763 s`
- Время оптимизированной версии: `0.001025 s`
- Ускорение: `921.96x`
- Пиковое потребление памяти у исходной версии: `360.54 KiB`
- Пиковое потребление памяти у оптимизированной версии: `10.21 KiB`
- Снижение потребления памяти: `35.33x`
